# Term Proxy Design Spec

Date: 2026-06-26

## Scope

Term Proxy 是一个使用 Tauri + React + Vite 构建的轻量级跨平台桌面工具，用于统一管理终端 Shell 代理环境变量。

MVP 明确只管理终端 Shell profile 中的代理环境变量，不直接修改系统级网络代理。系统代理能力通过接口预留，后续再按 macOS、Linux、Windows 分别实现。

## Product Reference

项目灵感来自 `farion1231/cc-switch`：https://github.com/farion1231/cc-switch

借鉴点：

- Tauri + React + TypeScript + Vite 的跨平台桌面工具栈。
- 前端通过 Tauri IPC 调用 Rust backend，后端按 Commands -> Services -> Data/Config 分层。
- shadcn/ui、react-i18next、多语言、主题、自启、跨平台打包意识。
- 配置管理中的单一事实源、原子写入、并发安全、导入/同步保护思路。

边界：

- 不复制 `cc-switch` 代码。
- 不做 AI provider、MCP、session、usage tracking、cloud sync 等非代理功能。
- 产品定位聚焦终端代理变量管理。

## User Requirements

已确认需求：

- 技术栈：Tauri + React + Vite，轻量级开发。
- UI：shadcn/ui 按需导入，简约精致，使用 taste-skill 审美约束。
- 工程质量：使用 impeccable 思路控制性能、a11y、响应式、状态和组件质量。
- 多语言：i18next + react-i18next，支持简体中文、英文、日语、繁体中文。
- 主题：浅色、深色、跟随系统。
- 系统设置：开机自动启动。
- 代理类型：`http_proxy`、`https_proxy`、`ALL_PROXY`。
- 启用规则：每种代理类型可保存多个配置，但同一类型同一时间只能启用一个；不同类型可同时启用。
- 环境变量大小写：MVP 只写 `http_proxy`、`https_proxy`、`ALL_PROXY`。
- no_proxy：全局设置，不放在每条代理配置上。
- 凭证：MVP 允许明文保存到 Tauri app config JSON，后续再迁移到系统凭据存储。
- Shell 支持：第一版实现 zsh、bash、PowerShell，内部通过 adapter 预留 fish、nu、cmd。
- profile 策略：profile 只插入受控 marker 和 source/import，实际代理内容写入托管文件。
- 启动读取：扫描用户现有 profile 中的代理配置，提示导入，不自动迁移。
- 跨平台：从第一版开始考虑 macOS、Linux、Windows 三端路径、换行、profile、打包差异。

## Architecture

采用平台适配层优先方案。

### Frontend

React 层只负责：

- UI 展示。
- 表单校验。
- 交互状态。
- 调用 typed Tauri API wrapper。
- i18n、主题、响应式布局。

React 不直接处理 profile 路径、文件写入、Shell 语法或平台权限。

建议目录：

```text
src/
  app/
  features/
    proxies/
    settings/
    import/
  shared/
    ui/
    i18n/
    tauri/
    lib/
    types/
```

### Backend

Rust/Tauri 层负责：

- 文件系统读写。
- profile marker 插入/移除。
- 托管脚本生成。
- 代理配置持久化。
- 启动扫描。
- 平台和 Shell 适配。
- 自启、主题系统跟随等桌面能力。

建议模块：

```text
src-tauri/src/
  commands/
  services/
  models/
  storage/
  shell/
  profile/
  import_scanner/
  system_proxy/
```

核心边界：

- `commands`：Tauri command API 层，只做参数校验和调用 service。
- `services`：业务编排，处理启用规则、写入流程、错误聚合。
- `storage`：读取和保存 app config JSON。
- `shell`：zsh、bash、PowerShell adapter。
- `profile`：受控 marker、托管文件、原子写入。
- `import_scanner`：读取 profile 中已有代理配置并返回候选。
- `system_proxy`：只定义 trait/interface，MVP 不修改系统代理。

## Data Model

```ts
type ProxyKind = "http_proxy" | "https_proxy" | "ALL_PROXY";

type ProxyScheme = "http" | "https" | "socks4" | "socks5";

type ProxyConfig = {
  id: string;
  name: string;
  kind: ProxyKind;
  scheme: ProxyScheme;
  host: string;
  port: number;
  username?: string;
  password?: string;
  enabled: boolean;
  createdAt: string;
  updatedAt: string;
};

type AppSettings = {
  theme: "light" | "dark" | "system";
  language: "zh-CN" | "en" | "ja" | "zh-TW" | "system";
  autoLaunch: boolean;
  noProxy: string;
  shellIntegration: {
    zsh: boolean;
    bash: boolean;
    powershell: boolean;
  };
};
```

Rust 侧定义对应 serde model。TypeScript 类型和 Rust model 必须保持字段语义一致。

## Enable Rules

启用某条代理时：

1. 按 `kind` 找到同类型配置。
2. 将同类型其他配置设为 `enabled = false`。
3. 将当前配置设为 `enabled = true`。
4. 保存配置 JSON。
5. 根据所有启用配置重新生成托管脚本。
6. 返回写入结果和提示：新终端会话生效，已有终端可能需要重新 source 或重启。

关闭某条代理时：

1. 将该配置设为 `enabled = false`。
2. 保存配置 JSON。
3. 重新生成托管脚本，移除对应变量。

禁用全部时清空所有启用状态，但保留 profile 集成 marker 和配置数据。

移除集成时从 profile 中删除受控 marker，并保留 app 配置 JSON。

## Profile Strategy

### macOS / Linux

支持：

- `~/.zshrc`
- `~/.bashrc`

profile 中只插入受控块：

```sh
# >>> term-proxy initialize >>>
[ -f "$HOME/.term-proxy/proxy.sh" ] && source "$HOME/.term-proxy/proxy.sh"
# <<< term-proxy initialize <<<
```

托管文件：

```text
~/.term-proxy/proxy.sh
```

托管文件示例：

```sh
# Generated by Term Proxy. Do not edit manually.
export http_proxy="http://127.0.0.1:1087"
export https_proxy="http://127.0.0.1:1087"
export ALL_PROXY="socks5://127.0.0.1:1080"
export no_proxy="localhost,127.0.0.1"
```

### Windows PowerShell

支持：

```text
~/Documents/PowerShell/Microsoft.PowerShell_profile.ps1
```

profile 中只插入受控块：

```powershell
# >>> term-proxy initialize >>>
$termProxyProfile = Join-Path $HOME ".term-proxy\proxy.ps1"
if (Test-Path $termProxyProfile) { . $termProxyProfile }
# <<< term-proxy initialize <<<
```

托管文件：

```text
~/.term-proxy/proxy.ps1
```

托管文件示例：

```powershell
# Generated by Term Proxy. Do not edit manually.
$env:http_proxy = "http://127.0.0.1:1087"
$env:https_proxy = "http://127.0.0.1:1087"
$env:ALL_PROXY = "socks5://127.0.0.1:1080"
$env:no_proxy = "localhost,127.0.0.1"
```

### Safety

- profile 修改必须通过 marker 定位，不得删除 marker 外的用户内容。
- 托管文件完整重写，使用临时文件 + rename 的原子写入策略。
- 生成脚本时必须对 URL、用户名、密码和路径做正确转义。
- Windows、macOS、Linux 的换行和路径分隔符必须由 adapter 决定。
- 关键跨平台逻辑使用中文注释解释原因。

## Import Scanner

启动时扫描支持的 profile：

- zsh: `~/.zshrc`
- bash: `~/.bashrc`
- PowerShell: `$PROFILE`

识别候选：

- `export http_proxy=...`
- `export https_proxy=...`
- `export ALL_PROXY=...`
- `$env:http_proxy = "..."`
- `$env:https_proxy = "..."`
- `$env:ALL_PROXY = "..."`

扫描结果只生成导入候选，不自动修改用户 profile。用户确认导入后新增 `ProxyConfig`。是否注释旧 profile 行必须单独确认，默认不改动。

## UI Design

主界面是实际工具，不做 landing page。

结构：

- 顶部状态栏：应用名、集成状态、当前语言、主题、设置入口。
- 类型切换：`http_proxy`、`https_proxy`、`ALL_PROXY`。
- 代理列表：显示名称、scheme、host、port、启用开关、编辑、删除。
- 编辑面板：名称、类型、scheme、host、port、用户名、密码。
- 设置页：主题、语言、开机自启、全局 `no_proxy`、Shell 集成状态、移除集成。
- 导入提示：发现现有代理后展示候选，用户确认导入。

状态：

- 未集成。
- 已集成。
- 部分集成。
- 写入失败。
- 需要重新打开终端。
- 有可导入配置。

视觉方向：

- 简约 product UI。
- restrained moss/olive 主色。
- 不用 AI 紫渐变、玻璃拟态、装饰性阴影。
- 卡片只用于条目和分组，禁止卡片套卡片。
- 路径和代理 URL 使用 mono 字体，支持复制和 tooltip。

## i18n

使用 `i18next` 与 `react-i18next`。

资源：

```text
src/shared/i18n/locales/
  zh-CN.json
  en.json
  ja.json
  zh-TW.json
```

规则：

- 所有可见文本必须走翻译 key。
- 错误信息、空状态、toast、表单校验都要国际化。
- 默认语言跟随系统。
- 用户手动选择后保存到 settings。

## System Settings

主题：

- `light`
- `dark`
- `system`

自启：

- 使用 Tauri 官方插件实现。
- 开启失败必须展示具体错误。
- UI 状态以系统实际结果为准，不做乐观假设。

## Testing

Rust 测试重点：

- proxy URL 生成和转义。
- 每种 `kind` 单选启用规则。
- zsh/bash/PowerShell 托管脚本生成。
- profile marker 插入、重复插入防护、移除。
- import scanner 解析。
- 原子写入错误处理。

前端测试重点：

- 代理表单校验。
- 类型切换和启用开关。
- 编辑、删除、禁用全部。
- 导入确认流程。
- 主题切换。
- 语言切换。
- 错误和空状态。

质量命令：

```bash
pnpm typecheck

pnpm build
cargo fmt --check
```

## Packaging

从第一版开始保留三端打包意识：

- macOS：`.dmg` / `.app`，后续考虑签名、公证、自启权限。
- Windows：`.msi` 或 NSIS，PowerShell profile 路径和执行策略要验证。
- Linux：`.deb`、`.rpm`、AppImage，Shell/profile 检测需适配主流发行版。

MVP 不要求完成签名和发布流水线，但项目结构不得阻碍后续三端打包。
